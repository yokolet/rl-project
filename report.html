Controlling the Quadcopter¶

We provide a sample agent in the code cell below to show you how to use the sim to control the quadcopter. This agent is even simpler than the sample agent that you'll examine (in agents/policy_search.py) later in this notebook!

The agent controls the quadcopter by setting the revolutions per second on each of its four rotors. The provided agent in the Basic_Agent class below always selects a random action for each of the four rotors. These four speeds are returned by the act method as a list of four floating-point numbers.

For this project, the agent that you will implement in agents/agent.py will have a far more intelligent method for selecting actions!

import random

class Basic_Agent():
    def __init__(self, task):
        self.task = task
    
    def act(self):
        new_thrust = random.gauss(450., 25.)
        return [new_thrust + random.gauss(0., 1.) for x in range(4)]

import csv
import numpy as np
from task import Task

# Modify the values below to give the quadcopter a different starting position.
runtime = 5.                                     # time limit of the episode
init_pose = np.array([0., 0., 10., 0., 0., 0.])  # initial pose
init_velocities = np.array([0., 0., 0.])         # initial velocities
init_angle_velocities = np.array([0., 0., 0.])   # initial angle velocities
file_output = 'data.txt'                         # file name for saved results

# Setup
task = Task(init_pose, init_velocities, init_angle_velocities, runtime)
agent = Basic_Agent(task)
done = False
labels = ['time', 'x', 'y', 'z', 'phi', 'theta', 'psi', 'x_velocity',
          'y_velocity', 'z_velocity', 'phi_velocity', 'theta_velocity',
          'psi_velocity', 'rotor_speed1', 'rotor_speed2', 'rotor_speed3', 'rotor_speed4']
results = {x : [] for x in labels}

# Run the simulation, and save the results.
with open(file_output, 'w') as csvfile:
    writer = csv.writer(csvfile)
    writer.writerow(labels)
    while True:
        rotor_speeds = agent.act()
        _, _, done = task.step(rotor_speeds)
        to_write = [task.sim.time] + list(task.sim.pose) + list(task.sim.v) + list(task.sim.angular_v) + list(rotor_speeds)
        for ii in range(len(labels)):
            results[labels[ii]].append(to_write[ii])
        writer.writerow(to_write)
        if done:
            break

The Task¶

A sample task has been provided for you in task.py. Open this file in a new window now.

The __init__() method is used to initialize several variables that are needed to specify the task.

  • The simulator is initialized as an instance of the PhysicsSim class (from physics_sim.py).
  • Inspired by the methodology in the original DDPG paper, we make use of action repeats. For each timestep of the agent, we step the simulation action_repeats timesteps. If you are not familiar with action repeats, please read the Results section in the DDPG paper.
  • We set the number of elements in the state vector. For the sample task, we only work with the 6-dimensional pose information. To set the size of the state (state_size), we must take action repeats into account.
  • The environment will always have a 4-dimensional action space, with one entry for each rotor (action_size=4). You can set the minimum (action_low) and maximum (action_high) values of each entry here.
  • The sample task in this provided file is for the agent to reach a target position. We specify that target position as a variable.

The reset() method resets the simulator. The agent should call this method every time the episode ends. You can see an example of this in the code cell below.

The step() method is perhaps the most important. It accepts the agent's choice of action rotor_speeds, which is used to prepare the next state to pass on to the agent. Then, the reward is computed from get_reward(). The episode is considered done if the time limit has been exceeded, or the quadcopter has travelled outside of the bounds of the simulation.

In the next section, you will learn how to test the performance of an agent on this task.

The Agent¶

The sample agent given in agents/policy_search.py uses a very simplistic linear policy to directly compute the action vector as a dot product of the state vector and a matrix of weights. Then, it randomly perturbs the parameters by adding some Gaussian noise, to produce a different policy. Based on the average reward obtained in each episode (score), it keeps track of the best set of parameters found so far, how the score is changing, and accordingly tweaks a scaling factor to widen or tighten the noise.

def running_mean(x, N):
    cumsum = np.cumsum(np.insert(x, 0, 0))
    return (cumsum[N:] - cumsum[:-N]) / N

def plot_rewards(reward_list):
    episodes, rewards = np.array(reward_list).T
    smoothed_rewards = running_mean(rewards, 10)
    plt.plot(episodes[-len(smoothed_rewards):], smoothed_rewards)
    plt.plot(episodes, rewards, color='gray', alpha=0.3)
    plt.xlabel('Episode')
    plt.ylabel('Average Reward')

Reflections¶

Question 1: Describe the task that you specified in task.py. How did you design the reward function?

Answer:

The biggest addition to the original reward function is a distance normalization. This is to easily limit the reward range to [-1, 1] (inclusive). However, actual maximum x, y and z values are unknown, so the normalization is not accurate for all positions. If the x, y and z are between initial and target positions, the normalized distance will have a good value. Once the position exceeds the range, the reward will be -1 to penalize the movement.

Another change is the reward curve. The calculation is done by average distance difference of the current position to the target position's x, y and z. To avoid steep curve, the task uses the reward function: 1 - (average difference of x, y and z)^0.4.

Additionally, only for z direction related rewards (penalties) were added. When z value (height) goes to negative, the reward will be -1. This is considered a crash. When z velocity is positive, the exponentially curved value wil be added up. On the contrary, z velocity is negative, the penalty of -1.0 will be added. The reason of the adjustment is the quadcoptor should go up.

Question 2: Discuss your agent briefly, using the following questions as a guide:

  • What learning algorithm(s) did you try? What worked best for you?
  • What was your final choice of hyperparameters (such as $\alpha$, $\gamma$, $\epsilon$, etc.)?
  • What neural network architecture did you use (if any)? Specify layers, sizes, activation functions, etc.

Answer:

- Learning Algorithm¶

I chose Actor-Critic Methods with Experience Replay and OrnsteinUhlenbeck noise. This algorithm appeared in the end of the lecture which doesn't have an implementation exercise. So, I decided to try the Actor-Critic Methods. Luckily, almost complete code for an actor, critic and agent are provided. I borrowed those in this project.

I learned the model is called DDPG (Deep Deterministic Policy Gradient), which is known to work with continuous input data (reference 4) like the quadcoptor. Both an actor and critic are neural networks. The actor works as a policy function to choose the best action from a continuous value space. The critic works as a value function to calculate a temporal difference (TD) error for the actor's policy function evaluation. The critic's output is a Q-value function.

Both actor and critic have two networks, actor_local/actor_target and ciritc_local/critic_target. Those exist to avoid a diverge caused by directly updating weights (reference 4).

The agent has a replay buffer as well. The buffer saves sets of state, action, reward, next_state, and done as experiences. While the learning process is going on, specified size of experiences are randomly chosen and used to avoid the correlations between episodes.

The last piece to mention about is the Ornstein-Uhlenbeck noise. The OpenAI blog post (reference 5) explains that adding noise "frequently boosts performance." The popular noise function is the Ornstein-Uhlenbeck noise (OU noise), which is a random walk type noise function. Since the randomness is defined by a Gaussian distribution for the OU noise, the randomized action (a set of rotor speeds) is within a moderated range.

An addition to the agent code provided is an epsilon decay to explore. The original code already has the Ornstein-Uhlenbeck noise (the idea of exploration). However, I added more to explore with random rotor speeds, 0 to 500, at the beginning.

- Hyperparameters¶

I tried different values of various parameters. It turned out many of them look good with the default values. The final value choices are:

- episodes (training): 2000
- learning_rate, actor: 0.0001
- learning_rate, critic: 0.001
- gamma (agent): 1.0
- tau (agent): 0.001
- epsilon (agent): 1.0
- epsilon decay (agent): 0.995

- Neural Network Architectures¶

The actor and critic models are the convolutional neural network (CNN).

- actor
    - 3 hidden layers and one output layer
    - activation function
        - ReLU for all hidden layers
        - Sigmoid for the output layer
    - optimizer: Adam optimizer with learning_rate 0.0001
- critic
    - 4 hidden layers and one output layer
    - activation function
        - ReLU for all hidden layers
        - None for the output layer
    - optimizer: Adam optimizer with learning_rate 0.001 (default)

As far as I look at the average reward plot, I don't think this is not easy to learn. The smoothed line (blue) shows much better curve; however, the actual average rewards line (gray) keeps bumpy and noisy by the time of 2000 episodes end.

This may have come from the Ornstein-Uhlenbeck noise added to the predicted action. The model may be sensitve to such noises even though the noise is in the moderated range. When I ran the training process multiple times repeatedly, I didn't get the same or similar reward curves. However, the reward function is a big area to be considered. When I used a different reward function, the learning curve was very poor, and the quadcoptor didn't fly mostly. Even a small change in the reward function showed the remarkable difference in the reward curve. Sometimes, the last 10 episodes showed good average rewards, but the curve had reatively big valley in the middle. The model may have predicted a good action, but probably, the reward function couldn't evaluate the changed state appropriately.

Finally, an overall performance looks not bad. If I look at plotted height, the quadcoptor mostly flies. The average rewards stay in the highest level after around 800 episodes.

The hyperparameter tuning is another hard part. I tried different values, then was back to the defaults for many parameters. My choice of the model, the Actor-Critic methods, looks right after I learned the model through the lecture and the blog post (reference 2, 4). The model seems brilliant. and works well out-of-the-box. The downside is the model is complex. Typical hyperparameters for the reinforcement learning didn't show much performance differences by changing those. It looks deeper insight of this complex model is required to tuning.

Simply, I found the reinforcement learning worked to fly quadcoptor. While going over the lecture, I tried some OpenAI toy text environments. It was fun, but the quadcoptor is more real and interesting.

The quadcoptor simulator code, physics_sim.py, is also interesting. It has many parameters and controls though I looked at and used only a small portion.
